# Supabase Setup — Accounts & Storage for BetterWords v3

*Implements Option 2 of `ACCOUNTS-PLAN.md` (Supabase Auth + Postgres with
row-level security), with the migrations-in-repo workflow. Everything is built
against the **v3** app only; v1/v2 are untouched.*

## What's already done (in this repo)

- `supabase/migrations/20260717000001_initial_schema.sql` — profiles, threads,
  messages, context_entries; owner-only RLS on every table; auto-created
  profile on signup; `plan_tier` groundwork (clients can't write it).
- `supabase/config.toml` — CLI project config (generated by `supabase init`).
- The Supabase CLI is a dev dependency — run it as `npx supabase …` or via the
  npm scripts below.
- v3 client integration (all under `src/v3/`):
  - `lib/supabase.js` — client seam; if the env keys are missing, v3 runs
    exactly as before (no accounts UI, nothing persisted).
  - `lib/auth.jsx` — email-OTP sign-in sheet (enter email → one-time code) +
    the header account control. Dev deep-link: add `&signin=1` to open the
    sheet.
  - `lib/db.js` — data layer: one thread per scenario run; draft versions and
    the sent letter are messages in it, with tone/length/strategy captured in
    each message's `context` and the clarify answers in the thread's.
  - Composer's **Save as New Draft** is the sign-in gate: compose freely
    anonymously, and the first save asks for an account. **Copy to clipboard**
    on the send screen records the sent letter for signed-in users
    (best-effort, never blocks the copy).

## What only you can do (one-time, ~10 minutes)

1. **Create the project.** supabase.com → New project. Pick the same AWS
   region as the Amplify app. Note the **project ref** (in the URL) and the
   database password you choose.

2. **Push the schema.**
   ```sh
   npx supabase login          # opens the browser
   npm run db:link -- --project-ref <your-project-ref>
   npm run db:push
   ```

3. **Make the sign-in email send a code.** Go straight to
   https://supabase.com/dashboard/project/_/auth/templates (the `_` resolves
   to your project), or navigate: left sidebar → **Authentication** (shield
   icon) → under *Configuration*, **Emails** (called *Email Templates* in
   older dashboard layouts; note it is NOT under Project Settings). Open the
   **Magic Link** template and replace the `{{ .ConfirmationURL }}` link with
   the one-time-code variable `{{ .Token }}`. The project's OTP length
   (Authentication → Providers → Email) is set to 6 digits to match the
   sign-in sheet's `maxLength`. E.g.:

   ```html
   <h2>Your BetterWords sign-in code</h2>
   <p>Enter this code in the app: <strong>{{ .Token }}</strong></p>
   <p>It expires in one hour. If you didn't request it, ignore this email.</p>
   ```

   The app's sign-in sheet asks for this code. Removing the link entirely is
   also Supabase's recommended fix for email providers that prefetch links
   and accidentally consume them.

4. **Wire the keys.** Dashboard → Settings → API: copy the Project URL and
   the `anon` public key into:
   - `.env` locally (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` — see
     `.env.example`), then restart `npm run dev`;
   - the Amplify console → Hosting → Environment variables (same two names)
     for the deployed build. Vite reads them at build time; no `amplify.yml`
     change needed.

That's it — the "Sign in" control appears in the v3 header once the keys are
present.

> **Production email note:** Supabase's built-in email sender works for solo
> development only: it sends at most 2 emails/hour AND only to members of
> your Supabase organization — anyone else gets "Email address not
> authorized" and no code. Before anyone besides you signs in, connect custom
> SMTP (Authentication → Emails → **SMTP Settings** tab) with a provider like
> Resend, Postmark, or AWS SES, using a domain you've verified with them.
> Custom SMTP starts capped at 30 emails/hour — raise it under
> Authentication → Rate Limits when needed.

## Day-to-day schema workflow

- New change: `npm run db:new -- <name>` → edit the generated file in
  `supabase/migrations/` → `npm run db:push`.
- Local database (optional, needs Docker Desktop): `npm run db:start` gives a
  full local Postgres + auth emulator with the migrations applied;
  `npm run db:stop` when done. Docker wasn't installed when this was set up,
  so migrations have been reviewed but first run against your hosted project.

## Data model recap

`profiles` (1:1 with auth.users, `plan_tier` for future paid tiers) →
`threads` (one per recipient/situation; clarify answers in `context` jsonb) →
`messages` (`kind`: draft_version / sent / reply / followup; tone & length in
`context` jsonb). `context_entries` is the append-only log for things the app
learns over time (user-level when `thread_id` is null). New context = new
jsonb keys, no migration; see the "Data model detail" section of
`ACCOUNTS-PLAN.md` for the reasoning.
